# RaceForgeCampaign 発表原稿

## 1枚目: 表紙

これから、Development Tool の課題として作成したスマートコントラクトについて発表します。

今回作ったものは、`RaceForgeCampaign` というレース車の強化をテーマにしたスマートコントラクトです。

内容としては、ユーザーが無料でレース車を作成できて、そのあと Sepolia ETH を支払うことで、カーブ制御、最高速度、加速度を強化できるというものです。

Webゲームの画面も作っていますが、提出の中心は Solidity のスマートコントラクトです。

## 2枚目: 課題要件の確認

この課題では、主に5つの要件を満たす必要があります。

1つ目は、外部から呼び出せる `external function` を作ることです。

2つ目は、コントラクトをデプロイしたウォレットだけが実行できる機能を、`modifier` を使って作ることです。

3つ目は、ETHを支払う必要がある `payable function` を作ることです。

4つ目は、操作結果を記録する `Event` を作ることです。

5つ目は、コントラクトを Sepolia テストネットにデプロイすることです。

今回の実装では、この5つをすべて満たすように作りました。

## 3枚目: テーマ説明

課題の例では、シンプルなポケモンがテーマになっていました。

ポケモンの名前、攻撃力、防御力があって、無料でポケモンを作成でき、有料で攻撃力や防御力を上げられるという内容です。

今回、私はこの構造をそのまま参考にして、テーマだけをレース車に変えました。

ポケモンの攻撃力や防御力にあたる部分を、レース車のカーブ制御、最高速度、加速度に置き換えています。

つまり、無料で車を作成できて、有料で能力値を上げて、その操作を Event で記録する、という形です。

## 4枚目: 要件1 external function

まず、要件1の `external function` について説明します。

コードでは、`createCar` と `recordRaceResult` という関数を `external` にしています。

`createCar` は、ユーザーが無料でレース車を作成するための関数です。

`recordRaceResult` は、レースの結果を記録するための関数です。

このように `external` と書いているので、コントラクトの外部から呼び出せる関数になっています。

そのため、要件1を満たしています。

## 5枚目: 要件2 modifier

次に、要件2のデプロイ者だけが実行できる機能について説明します。

コードでは、`onlyDeployer` という `modifier` を作っています。

この modifier の中で、`msg.sender == deployer` かどうかを確認しています。

`msg.sender` は、この関数を呼び出したウォレットアドレスです。

そして `deployer` は、コントラクトを最初にデプロイしたウォレットアドレスです。

つまり、デプロイした人と実行しようとしている人が同じかどうかをチェックしています。

この `onlyDeployer` を `getContractBalance` に付けることで、コントラクト残高はデプロイ者だけが確認できるようにしました。

そのため、要件2を満たしています。

## 6枚目: 要件3 payable function

次に、要件3の `payable function` について説明します。

今回のコントラクトでは、`upgradeControl`、`upgradeSpeed`、`upgradeAccel` という3つの強化関数を作りました。

それぞれ `payable` を付けているので、ETHを受け取れる関数になっています。

また、`UPGRADE_FEE` を `0.003 ether` に設定していて、`require` で支払い額がちょうど `0.003 ETH` かどうかを確認しています。

これによって、ETHを支払ったときだけ能力値を強化できるようにしています。

そのため、要件3を満たしています。

## 7枚目: 要件4 Event

次に、要件4の Event について説明します。

今回の実装では、主に `CarUpgraded` と `RaceResultRecorded` という Event を作っています。

`CarUpgraded` は、ユーザーが車の能力値を強化したときに発行されます。

どのユーザーが、どの車の、どの能力値を、何レベルに上げたかを記録します。

また、`address indexed user` としているので、ユーザーのウォレットアドレスで検索しやすくなっています。

`RaceResultRecorded` は、レース結果を記録する Event です。

このように、操作結果を Event として残しているので、要件4を満たしています。

## 8枚目: 要件5 Sepolia デプロイ

次に、要件5の Sepolia テストネットへのデプロイについてです。

作成した `RaceForgeCampaign` コントラクトは、Sepolia テストネットにデプロイしました。

このスライドに、デプロイ済みのコントラクトアドレスと、デプロイトランザクションを載せています。

これによって、ローカルで作っただけではなく、実際にテストネット上に公開できていることが確認できます。

そのため、要件5も満たしています。

## 9枚目: 実行結果

この画面は、今回のテーマをわかりやすく見せるために作った Webゲームのデモです。

自分の車は `YOU` と表示され、カーブ制御、最高速度、加速度を強化できるようになっています。

レースは1周で、他のAI車両と順位を競う形にしています。

ただし、この Webゲーム自体が課題の必須条件というよりは、スマートコントラクトで作ったテーマを説明しやすくするためのものです。

実際に ETH を支払う必要がある部分は、Solidity 側の `payable function` で実装しています。

## 10枚目: まとめ・感想

最後にまとめです。

今回の課題では、`external function`、`modifier`、`payable function`、`Event`、そして Sepolia へのデプロイという5つの要件を満たすように実装しました。

テーマは先生のポケモン例を参考にして、レース車の能力値強化に置き換えました。

最初は、`payable` や `modifier` がどこで必要になるのか少し分かりにくかったです。

しかし、無料で作成する、ETHを払って強化する、結果を Event で記録する、という流れにすると、それぞれの役割が理解しやすくなりました。

次にやるなら、Web画面から直接 MetaMask を使って、コントラクトの関数を呼び出せるようにしたいです。

以上で発表を終わります。
