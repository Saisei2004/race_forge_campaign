# Iot特論 Blockchain 発表台本

## 1ページ目: 表紙

これから、Development Tool の課題として作成したスマートコントラクトについて発表します。

今回作成したものは、`RaceForgeCampaign` というスマートコントラクトです。

テーマは、レース車を作って、Sepolia ETH を支払うことで車の能力値を強化できる、というものです。

具体的には、Solidity でスマートコントラクトを作成し、それを Sepolia テストネットにデプロイしました。

また、スマートコントラクトの内容をイメージしやすくするために、Webゲームの画面も作成しました。

この発表では、課題要件、テーマ、コード、実行結果、最後に感想の順番で説明します。

## 2ページ目: 課題要件

まず、今回の課題要件について説明します。

この課題では、主に5つの要件を満たす必要がありました。

1つ目は、外部から呼び出せる `external function` を1つ以上作ることです。

2つ目は、コントラクトをデプロイしたウォレットアドレスだけが実行できる機能を、`modifier` を使って作ることです。

3つ目は、ETHを支払う必要がある `payable function` を1つ以上作ることです。

4つ目は、`Event` を実装して、操作結果やトランザクション結果を記録することです。

5つ目は、作成したコントラクトを Sepolia テストネットにデプロイすることです。

今回の実装では、この5つの要件をすべて満たしています。

## 3ページ目: テーマ説明

次に、今回のテーマについて説明します。

課題の例では、シンプルなポケモンがテーマになっていました。

ポケモンには、名前、攻撃力、防御力があり、無料でポケモンを作成できて、有料で攻撃力や防御力を上げるという内容でした。

今回、私はこの構造を参考にして、テーマをレース車に置き換えました。

ポケモンの攻撃力や防御力にあたる部分を、レース車のカーブ制御、速度、加速度にしました。

つまり、無料でレース車を作成できて、ETHを支払うことで能力値を1段階ずつ強化できる、という仕組みです。

また、強化した操作やレース結果を Event として記録するようにしています。

## 4ページ目: 要件1 external function

ここからは、課題要件に対応するコードについて説明します。

まず、要件1の `external function` です。

コードでは、`createCar` という関数を `external` にしています。

この関数は、ユーザーが無料でレース車を作成するためのものです。

また、`recordRaceResult` という関数も `external` にしています。

これは、レース結果を記録するための関数です。

`external` と書くことで、コントラクトの外部から呼び出せる関数になります。

そのため、この部分で要件1を満たしています。

## 5ページ目: 要件2 modifier

次に、要件2の、デプロイ者だけが実行できる機能について説明します。

コードでは、`onlyDeployer` という `modifier` を作成しました。

この中で、`msg.sender == deployer` かどうかを確認しています。

`msg.sender` は、関数を呼び出したウォレットアドレスです。

`deployer` は、コントラクトをデプロイしたウォレットアドレスです。

つまり、この modifier では、実行しようとしている人がデプロイ者本人かどうかを確認しています。

そして、`getContractBalance` という関数に `onlyDeployer` を付けています。

これによって、コントラクトの残高確認は、デプロイしたウォレットだけが実行できるようになっています。

この部分で、要件2を満たしています。

## 6ページ目: 要件3 payable function

次に、要件3の `payable function` について説明します。

今回のコントラクトでは、車の能力値を強化する関数を3つ作りました。

`upgradeControl` はカーブ制御を強化する関数です。

`upgradeSpeed` は速度を強化する関数です。

`upgradeAccel` は加速度を強化する関数です。

この3つの関数には、すべて `payable` を付けています。

`payable` を付けることで、その関数は ETH を受け取れるようになります。

また、`UPGRADE_FEE` を `0.003 ether` に設定し、`require` で支払い額を確認しています。

そのため、0.003 ETH を支払ったときだけ、能力値を強化できるようになっています。

この部分で、要件3を満たしています。

## 7ページ目: 要件4 Event

次に、要件4の Event について説明します。

今回のコードでは、`CarUpgraded` と `RaceResultRecorded` という Event を作成しました。

`CarUpgraded` は、ユーザーが車の能力値を強化したときに発行される Event です。

どのユーザーが、どの車を、どの能力値について、何レベルに強化したかを記録します。

また、`address indexed user` としているので、ユーザーのウォレットアドレスで検索しやすくなっています。

`RaceResultRecorded` は、レース結果を記録するための Event です。

このように、ユーザーの操作結果をブロックチェーン上に記録できるようにしているので、要件4を満たしています。

## 8ページ目: 要件5 Sepoliaデプロイ

次に、要件5の Sepolia テストネットへのデプロイについて説明します。

作成した `RaceForgeCampaign` コントラクトは、Sepolia テストネットにデプロイしました。

このページには、デプロイしたコントラクトアドレスと、デプロイトランザクションを載せています。

これによって、ローカルでコードを書いただけではなく、実際にテストネット上に公開できていることが確認できます。

そのため、要件5も満たしています。

## 9ページ目: 実行結果

次に、実行結果について説明します。

この画面は、今回のテーマを分かりやすく見せるために作成した Webゲームのデモ画面です。

自分の車は `YOU` と表示されます。

また、Control、Speed、Accel の3種類を強化できるようにしています。

レースは1周で、AIの車と順位を競う形にしています。

ただし、このWebゲームは、課題の必須要件そのものというより、スマートコントラクトのテーマを見せるためのデモです。

実際にETHを支払う必要がある部分は、Solidity側の `payable function` で表現しています。

## 10ページ目: まとめ・感想

最後に、まとめと感想です。

今回の課題では、`external function`、`modifier`、`payable function`、`Event`、そして Sepolia へのデプロイという5つの要件を満たすように実装しました。

テーマは、先生のポケモン例を参考にして、レース車の強化という形に置き換えました。

最初は、仮想通貨とは何なのか、スマートコントラクトで何ができるのかを何となくしか理解できていませんでした。

しかし、今回の開発を通して、ETHをただ送るだけではなく、サービスの中の機能として使えることが分かりました。

特に、`payable function` によって支払いが必要な機能を作れることや、`Event` によってユーザーの操作を記録できることが印象に残りました。

この仕組みを使えば、ゲームやWebサービスの中で、支払いと機能を結びつけた開発ができると感じました。

今後は、Web画面から直接 MetaMask を使って、コントラクトを操作できるようにしてみたいです。

以上で発表を終わります。
